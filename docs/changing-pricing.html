TalvexIT — Changing subscription pricing

v1.0 · 2026-05-09 · Companion to docs/customer-subscription-plan.html + docs/supplier-subscription-plan.html
TL;DR. Edit prices/limits/flags in packages/shared/src/subscription-config.ts, then re-run pnpm --filter @onys/api seed:subscriptions. That writes to the DB and (if STRIPE_SECRET_KEY is set) syncs Stripe Products and Prices in the same step. The public /pricing page reads from the DB via GET /api/v1/subscriptions/plans and re-renders within the 1-hour Next.js revalidate window.

§1 The data flow

Top to bottom, where pricing data lives and who reads it:

1

Source of truth — packages/shared/src/subscription-config.ts

TypeScript constants. Two exported maps: CUSTOMER_PLANS (4 plans) and SUPPLIER_PLANS (4 plans). Each plan has price, trial days, limits, feature flags, sort order, CTA copy. This is the file that gets edited — everything below reads from here.

2

Seed script — apps/api/src/scripts/seed-subscription-plans.ts

Reads the two PLAN_LISTs above, maps each shared PlanDef shape into the Prisma upsert payload, and writes to subscription_plans via prisma.subscriptionPlan.upsert(where: { slug }, …). Idempotent — safe to re-run any time. If STRIPE_SECRET_KEY is set it also creates/updates Stripe Products + Prices and writes the IDs back to each row.

3

Database — SubscriptionPlan table

The runtime source. Every quota check, every public-pricing render, every POST /subscribe reads from here. Columns mirror the shared PlanDef shape one-to-one (monthly_price_aud, yearly_price_aud, max_active_tasks, allow_priority_listing, etc. — see prisma/schema.prisma line ~2572).

4

Public API — GET /api/v1/subscriptions/plans

Returns plans where is_active = true AND is_public = true, ordered by sort_order. Backed by SubscriptionService.getPublicPlans() in apps/api/src/services/subscription.service.ts.

5

Pricing page server fetch — apps/web/src/app/pricing/page.tsx

Server-renders the plans pre-fetched from the API with a revalidate: 3600 (1 hour) Next.js cache hint. The PricingClient component refetches client-side on mount too, so admin edits at runtime show up within seconds in a fresh tab.

6

Card rendering — apps/web/src/app/pricing/PricingClient.tsx

The PublicPlan interface mirrors the SubscriptionPlan columns. Cards render audience-aware feature lists from the plan's limits + flags. Don't edit prices here — this file only displays them.

§2 Three places to change pricing — which to use

Path Where Best for Survives re-seed? Stripe sync?
A · Recommended subscription-config.ts → run seed Permanent changes that should be in version control (price changes, new plans, retiring plans, limit revisions) ✓ It is the source ✓ Automatic on re-seed
B · Runtime tweaks /admin/subscriptions UI One-off ad-hoc adjustments — e.g. temporary promo on a single tier, fixing a typo in description without a deploy ✗ Overwritten Manual (click "Sync Stripe" per plan)
C · Emergencies only Direct SQL UPDATE Pre-deploy hotfix when the seed can't run — you should still update subscription-config.ts in the same PR so it stays the source of truth ✗ Overwritten No (do it manually in Stripe Dashboard)
Why Path A is the default. The seed is idempotent and runs on deploy. Any change made via Path B or C will silently revert the next time seed:subscriptions runs. Treat the admin UI as a "view-and-tweak" surface, not a source of truth.

3.2 Anatomy of a plan entry

'customer-business': {
  slug:                    'customer-business',     // immutable — DB key
  name:                    'Business',              // shown on the card
  plan_type:               'CUSTOMER_BUSINESS',     // enum, do not change
  description:             'For small teams …',     // 1-2 sentence pitch
  price_aud_monthly:       29,                      // ← edit prices here
  price_aud_yearly:        290,                     //   convention: 10×monthly
  trial_days:              14,                      // 0 for free plans
  auto_activate_on_signup: false,                   // exactly one per audience
  badge_text:              undefined,               // e.g. 'Most Popular'
  cta_text:                'Start Trial',           // button label
  highlight_color:         '#14b8a6',               // accent stripe hex
  sort_order:              20,                      // display order on /pricing
  limits: {                                         // ← edit quotas here
    task_bookings:   5,
    active_orders:   2,
    orders:          5,
    ai_scopes:       1,
    contracts:       1,
    active_tenders:  1,
  },
}

Supplier plans add a feature_flags object alongside limits:

feature_flags: {
  allow_priority_listing: true,
  allow_project_mode:     true,
  // … any subset of the 12 supplier flags
},

3.3 Procedure

  1. Open packages/shared/src/subscription-config.ts.
  2. Edit the plan you want. Save.
  3. Rebuild the shared package — the seed imports from @onys/shared's built dist, not the TS source: 
    pnpm --filter @onys/shared build
    Footgun. Skipping this step will silently run the seed against the previously-built values — the seed will report "upserted" but the DB ends up with the old numbers. Verify with the curl check in §9 if the price doesn't update.
  4. Run the seed: 
    pnpm --filter @onys/api seed:subscriptions
  5. Watch the output. You'll see [seed-plans] upserted: customer-business for each changed row, and [seed-plans] stripe synced: … if Stripe is enabled.
  6. Reload /pricing — the client-side refetch on mount picks up the new prices within seconds. If you're testing across server-rendered visits, wait for the Next.js revalidate: 3600 window or restart the web dev server.
  7. Commit the changed file. Don't commit just the DB — the source of truth is the file.
The seed is idempotent. It uses Prisma upsert on the slug column, so re-running doesn't duplicate rows. It also marks plans absent from the source as is_active=false, is_public=false (soft-retire). Run it as often as you like during iteration.

§4 Path B — /admin/subscriptions

For ad-hoc edits that don't justify a deploy. Available to PLATFORM_ADMIN with the manage_platform_config permission.

4.1 What the UI can do

ActionAPI endpointSurvives re-seed?
List all plans (incl. inactive)GET /admin/subscriptions/plansn/a
Create a one-off planPOST /admin/subscriptions/plans✗ — not in seed source
Update a plan (price, limits, flags, copy)PUT /admin/subscriptions/plans/:id✗ — overwritten
Soft-delete (set is_active=false)DELETE /admin/subscriptions/plans/:id✗ — seed re-activates if still in config
Sync a plan to StripePOST /admin/subscriptions/plans/:id/sync-stripen/a

4.2 Steps

  1. Visit /admin/subscriptions while signed in as PLATFORM_ADMIN.
  2. Click the plan row you want to edit → the PlanFormModal opens.
  3. Change prices, limits, feature flags, descriptions. Save.
  4. If you changed prices, click Sync to Stripe on that row — otherwise checkout will charge the old Stripe Price.
Drift warning. A change here is not reflected in subscription-config.ts. The next time anyone runs the seed (CI deploy, dev's laptop, anything) your change disappears. If the change is meant to be permanent, also edit the source file in the same PR.

§5 Path C — direct DB update

Use only when neither A nor B is feasible — for example: pre-launch promo at 3am and the build is broken. Same drift caveat as Path B, with extra risk because there's no admin validation.

-- Example: drop the Solo Pro monthly price from $29 to $19 for a weekend promo
UPDATE subscription_plans
SET monthly_price_aud = 19,
    yearly_price_aud  = 190
WHERE slug = 'supplier-solo';
Stripe will be out of sync. Direct DB updates do not touch Stripe Products/Prices. New subscribers will be charged the old Stripe Price unless you also:
  1. Create a new Price in the Stripe Dashboard manually, and
  2. Run POST /admin/subscriptions/plans/:id/sync-stripe for that plan ID.
And of course update subscription-config.ts in the same PR so the seed doesn't revert your change.

§6 Common scenarios

6.1 Change a plan's monthly price

  1. Open subscription-config.ts, find the plan map entry.
  2. Edit price_aud_monthly (and price_aud_yearly — convention is 10× monthly).
  3. Run pnpm --filter @onys/api seed:subscriptions.
  4. Verify on /pricing (client-side refetch — should be immediate in a fresh tab).
  5. Verify the new Stripe Price exists: Stripe Dashboard → Products → pick the plan → Prices.

6.2 Change a quota limit

  1. Edit the relevant key inside limits: { … }.
  2. Re-seed.
  3. The change is immediately enforced on quota gates — e.g. a customer who's already placed 4 orders on a plan whose orders limit just went from 5 to 3 will hit the 4th order through, then be blocked on the 5th with an upgrade prompt.

6.3 Add a feature flag to an existing plan

  1. In feature_flags: { … } on the supplier plan, set allow_priority_listing: true (or any of the 12).
  2. Re-seed. The flag column flips to true in the DB row.
  3. /pricing card immediately starts showing the corresponding feature label (e.g. "Priority placement in search").
  4. Anywhere in the app that gates on the flag (search ranking, analytics dashboards, etc.) starts honouring it on the next request.

6.4 Add a brand-new plan

  1. Add a new entry to either CUSTOMER_PLANS or SUPPLIER_PLANS in subscription-config.ts. Pick a unique slug.
  2. Pick a unique plan_type enum value — you'll need to add it to the PlanType enum in prisma/schema.prisma and the type union in subscription-config.ts, then generate + run a migration: 
    pnpm --filter api prisma migrate dev --name add_plan_type_xyz
  3. Re-seed. The plan is upserted with active+public flags = true and synced to Stripe.

6.5 Retire a plan

  1. Delete its entry from the plan map in subscription-config.ts.
  2. Re-seed. The seed sees the plan is absent from the source and runs: 
    UPDATE subscription_plans
SET is_active = false, is_public = false
WHERE slug = 'the-retired-slug';
  3. Existing subscribers keep their subscriptions until they cancel or renewal fails (Stripe Price remains valid).
  4. The plan disappears from /pricing immediately (filter is is_active && is_public).

§7 Stripe pricing-sync caveats

The seed creates Stripe resources in this order, per plan:

  1. A Stripe Product (named after the plan). Idempotent on plan.stripe_product_id — if already set, the Product is updated, not recreated.
  2. A monthly Price for the AUD monthly amount. Stored back to plan.stripe_price_id_monthly.
  3. A yearly Price for the AUD yearly amount. Stored back to plan.stripe_price_id_yearly.
Prices are immutable in Stripe. If you change a plan's monthly price, the seed creates a new Price ID and writes it to stripe_price_id_monthly. Existing subscribers stay on the old Price until they cancel and resubscribe. The old Price is archived — not deleted — on the Stripe side.
ScenarioWhat happens
Price change via Path ANew Stripe Price minted, old archived, new subscribers charged the new amount, old subscribers grandfathered.
Price change via Path B (UI)DB updated; Stripe untouched until you click "Sync Stripe" on that plan.
Price change via Path C (SQL)DB updated; Stripe untouched. You must create a Price in Stripe and run the sync endpoint manually.
Plan retired (Path A)DB row deactivated; Stripe Product remains (just hidden from public catalog). Existing subscribers continue billing on the existing Price.
Plan retired (Path B/C without seed alignment)Same as above, but a future seed will re-activate the plan — remove it from subscription-config.ts too.

§8 Cache — why your change isn't showing yet

Three places the new value might be stuck:

LayerWhat it cachesHow to force a refresh
Next.js server render The plans JSON pre-fetched in pricing/page.tsx — revalidate: 3600 (1 hour). Wait an hour, restart the dev server (pnpm dev), or trigger a new deploy in production.
PricingClient mount fetch The plans state after the client-side refetch. Lives only in the current React tree. Reload the page in a fresh tab — the on-mount fetch hits the live API and replaces whatever the server pre-rendered.
SubscriptionService config cache Limit-check code reads plan.limits.* via Prisma per-request — no in-process cache. n/a (changes are reflected immediately).
Quick check. Open dev tools, look at the response of GET /api/v1/subscriptions/plans in the Network tab. If the JSON shows your new values, the API is updated — any UI mismatch is one of the cache layers above.

§9 Verifying a change went through

In order of cost:

  1. Hit the API directly — fastest: 
    curl http://localhost:4000/api/v1/subscriptions/plans | jq '.data[] | {slug, monthly_price_aud, max_active_tasks}'
  2. Reload /pricing in a fresh tab. Toggle the audience tab — if your edit was to a supplier plan you won't see it on the customer tab.
  3. Query the DB directly: 
    cd apps/api && pnpm prisma studio
# or
psql $DATABASE_URL -c "SELECT slug, monthly_price_aud, yearly_price_aud FROM subscription_plans WHERE is_active;"
  4. Check Stripe — Stripe Dashboard → Products. Each plan should have exactly two active Prices (monthly + yearly) and any older Prices should be archived.

§10 Anti-patterns

Don't…Because…
Hard-code a price in PricingClient.tsx or any other UI file. The UI reads from the API, which reads from the DB, which is seeded from subscription-config.ts. Editing the UI bypasses every quota check that uses the DB row's price.
Skip the seed after editing subscription-config.ts. Nothing reads subscription-config.ts at runtime — the seed is the only path from the source file to the DB.
Edit only via /admin/subscriptions and forget to also update the source. The next seed run silently reverts your change. Always treat the admin UI as a "tweak-and-also-update-source" surface, not a permanent edit point.
Change a price via SQL UPDATE without creating a new Stripe Price. Checkout will charge the old Stripe Price. The plan card will say one number; the customer's card statement will show another.
Reuse a slug after retiring a plan. slug is the upsert key. A new plan with the same slug will update the retired row, including reactivating it. Pick a new slug.
Add a new plan_type enum value to subscription-config.ts without a Prisma migration. The seed will fail with a Prisma validation error because the enum doesn't accept your new value. Always migrate the schema first.
