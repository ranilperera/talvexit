A walk-through of how supplier → customer invoicing handles GST, cross-border supply, and offline-payment confirmation across every supplier/customer permutation. Audit + recommendations — revision 2026-05-08 (rev 1.1: P1 recommendations implemented).

1. Database fields that identify tax residency

Both User (used for individual contractors and customers) and ConsultingCompany carry the same set of fields:

FieldDefaultUsed in GST decision?Used for
billing_country 'AU' YES Source of customerCountry / providerCountry. Drives ABN-required check and cross-border detection. The single signal that controls everything.
tax_residency_country 'AU' NO Stored separately for display + reporting only.
is_foreign_entity false NO Set by the registration UI based on country choice. Decorative today.
gst_registered false YES (suppliers only) Drives providerGstRegistered. Customer's flag is correctly ignored — under AU GST law, the supplier's status determines whether GST is charged.
gst_registered_verified (User) / gst_registered_confirmed_at (Company) false / null NO Audit trail set when ABR confirms via the verify-ABN flow.
abn null indirectly Required for AU customers (server-side check). Required for AU suppliers to be GST-eligible.
abn_verified + abn_verified_name + abn_verified_at false / null NO Locks ABR-derived fields (legal name, GST flag, entity type, ACN).
vat_number null NO Stored for overseas tax IDs but not used in any decision.
Effective tax-residency identification today: only billing_country matters. Everything else is documentation — the system branches on that one field. tax_residency_country, is_foreign_entity, and vat_number are stored but unread.

2. The decision matrix

The supplier "type" collapses to three tax classifications — individual vs company makes no difference to GST:

Supplier classificationbilling_countrygst_registered
AU + GST'AU'true
AU + no GST'AU'false
Overseasnot 'AU'usually false

The decision pipeline (in raiseInvoice at tender-contract-payment.service.ts:108-140):

isCrossBorder = providerCountry ≠ customerCountry
                AND (providerCountry ≠ 'AU' OR customerCountry ≠ 'AU')

chargeGst     = providerGstRegistered AND NOT isCrossBorder

gstAmount     = chargeGst ? round(amountExGst × 0.10, 2 dp) : 0

Then computeGstTreatmentReason() returns one of these strings, in priority order:

#ConditionReturned text
1gst_charged"GST 10% applied"
2!issuerIsAu"No GST — overseas supplier (not subject to Australian GST)"
3!issuer_gst_registered"GST not applicable — supplier is not registered for GST"
4cross-border && recipient ≠ AU"GST-free export of services (s38-190 of the GST Act)"
5cross-border && recipient = AU (currently unreachable — see §5)"Reverse-charge applies — recipient liable for GST"
6catch-all"GST-free supply"

3. What happens in each scenario

Customer GST registration does not affect any branch — the rules below are identical whether the customer is GST-registered or not. So the original Cases 1 and 2 collapse to a single row.

AU AU + no GST × none INVOICE GST not applicable — supplier is not registered for GST

Overseas AU + GST × none TAX INVOICE GST-free export of services (s38-190 GST Act)

! AU customer + overseas supplier: should mention reverse-charge but currently doesn't (see §5).   ! Overseas customer + AU supplier without GST: doesn't differentiate from a domestic non-GST supply — cosmetic, not legally wrong.

Individual vs company doesn't affect GST treatment in any scenario — both paths use the same supplier gst_registered + billing_country fields.

4. Offline payment confirmation flow

Once the invoice is raised, the customer pays out-of-band (bank transfer, Wise, PayPal, etc.) and reports the payment back to TalvexIT. The supplier then verifies receipt against their bank account and either confirms or disputes.

StepStatus transitionAPIPersisted
1. Supplier raises invoice SENT (legacy) or AWAITING_PAYMENT (direct-payment) POST /provider/tender-contracts/:id/milestones/:mid/invoice Invoice row + PDF in blob
2. Customer pays offline + reports AWAITING_PAYMENT → PAYMENT_REPORTED POST /tender-contract-invoices/:id/payment/report payment_method, payment_reference, payment_amount_reported_aud, customer_reported_paid_at, payment_evidence_blob_path, evidence appended to payment_evidence_history[]
3a. Supplier confirms PAYMENT_REPORTED → PAID POST /tender-contract-invoices/:id/payment/confirm supplier_confirmed_paid_at, paid_at, latest history row marked CONFIRMED. PDF regenerated with PAID badge. Customer-copy + supplier-receipt emails queued.
3b. Supplier disputes PAYMENT_REPORTED → AWAITING_PAYMENT POST /tender-contract-invoices/:id/payment/dispute payment_dispute_reason, payment_dispute_raised_at, latest history row marked REJECTED. Customer can re-upload evidence (loops back to step 2).
The confirmation step does not check the bank account programmatically. The supplier eyeballs their bank statement, then clicks Confirm or Dispute. payment_evidence_history is append-only so a paper trail of every report → decision cycle is preserved. Open Banking integration would be a natural follow-up.

5. Maintainability gaps & risks

What's good today

  • Single canonical text helper. computeGstTreatmentReason() in invoice-template.ts centralises the "why no GST?" copy. Adding a new GST scenario means editing one function.
  • Treatment text stamped on the invoice row at creation (gst_treatment_reason column) so future re-renders never drift even if supplier flags change.
  • Cross-border + GST charge logic is small and self-contained inside raiseInvoice — about 30 lines, easy to read top-to-bottom.
  • Append-only payment evidence history with status flags (PENDING / CONFIRMED / REJECTED) keeps the dispute paper trail intact.

6. Prioritised recommendations

P1 DONE R1. Fix the dead reverse-charge branch

Implemented in commit 67b472c. Re-ordered computeGstTreatmentReason() in apps/api/src/utils/invoice-template.ts so the cross-border + recipient checks run before the issuer-overseas blanket return. The function as it now stands:

function computeGstTreatmentReason(args) {
  if (args.gst_charged) return 'GST 10% applied';

  const issuerIsAu    = args.issuer_country === 'AU' || args.issuer_country === null;
  const recipientIsAu = args.recipient_country === 'AU' || args.recipient_country === null;

  // Cross-border first — the relationship-level cases need to win.
  if (args.is_cross_border &&  recipientIsAu) {
    return 'Reverse-charge may apply — AU recipient liable for GST under Div 84';
  }
  if (args.is_cross_border && !recipientIsAu) {
    return 'GST-free export of services (s38-190 of the GST Act)';
  }
  if (!issuerIsAu) {
    return 'No GST — overseas supplier (not subject to Australian GST)';
  }
  if (!args.issuer_gst_registered) {
    return 'GST not applicable — supplier is not registered for GST';
  }
  return 'GST-free supply';
}

Verified output for AU customer + overseas supplier: "Reverse-charge may apply — AU recipient liable for GST under Div 84".

P1 DONE R2. Extract decideGstTreatment() into the shared template

Implemented in commit 67b472c. Both invoice flows now call decideGstTreatment() from apps/api/src/utils/invoice-template.ts — no duplicated GST logic, no drift risk. The shipped function:

// In utils/invoice-template.ts
export const AU_GST_RATE = 0.10;

export interface GstDecisionInput {
  issuer_country: string | null;
  issuer_gst_registered: boolean;
  recipient_country: string | null;
  amount_ex_gst_cents: number;
}
export interface GstDecision {
  charge_gst: boolean;
  gst_rate: number;            // 0.0 or 0.10
  gst_amount_cents: number;
  is_cross_border: boolean;
  is_tax_invoice: boolean;     // controls "TAX INVOICE" vs "INVOICE"
  treatment_reason: string;
}

export function decideGstTreatment(args: GstDecisionInput): GstDecision {
  const isCrossBorder =
    args.issuer_country !== null && args.recipient_country !== null &&
    args.issuer_country !== args.recipient_country &&
    (args.issuer_country !== 'AU' || args.recipient_country !== 'AU');

  const chargeGst = args.issuer_gst_registered && !isCrossBorder;
  const gstRate = chargeGst ? AU_GST_RATE : 0;
  const gstCents = Math.round(args.amount_ex_gst_cents * gstRate);

  return {
    charge_gst: chargeGst,
    gst_rate: gstRate,
    gst_amount_cents: gstCents,
    is_cross_border: isCrossBorder,
    is_tax_invoice: args.issuer_gst_registered && !isCrossBorder,
    treatment_reason: computeGstTreatmentReason({
      ...args,
      is_cross_border: isCrossBorder,
      gst_charged: chargeGst,
    }),
  };
}

Call sites:

  • apps/api/src/services/tender-contract-payment.service.ts → raiseInvoice() (~25 lines of inline math collapsed to one call).
  • apps/api/src/services/service-invoice.service.ts → computeTotals(). Now takes issuerCountry + recipientCountry and delegates the rate decision. New private helpers resolveRecipientCountry / resolveIssuerCountry on the class fetch billing_country from the parties.

Side-effect bug fix: the service-invoice flow previously charged 10% GST for any AU GST-registered supplier regardless of customer country. An AU consultant billing a US customer was incorrectly being charged GST. Now correctly emits GST-free export of services (s38-190 GST Act) with no GST.

The explicit tax_rate override path is preserved for admin-driven custom-rate invoices.

P2 R3. Make tax_residency_country meaningful or remove it

Two options:

  1. Wire it in: rename billing_country → billing_address_country (where invoices are sent), and use tax_residency_country for the GST decision. This better reflects ATO terminology — supply is taxed where the supplier is resident, not where invoices are addressed.
  2. Drop it: remove the column from schema, types, and registration UI. Less code is better than ambiguous code.

Recommendation: option 2 for the current direct-payment model. The platform doesn't need to model a non-AU resident supplier with an AU billing address — we don't have that case in the wild and it complicates everything. Drop the unused fields.

P2 R4. Surface customer GST status in BILL TO block

When the customer is GST-registered (AU + has ABN), render their ABN under their name on the invoice. Already partially done — just confirm the field is read. Helps the customer's AP team match the invoice and confirms ITC eligibility for the supplier.

P3 DONE R5. Constants over magic literals

Implemented in commit 67b472c alongside R2. AU_GST_RATE = 0.10 is now exported from invoice-template.ts and re-exported from service-invoice.service.ts. Both raiseInvoice() and computeTotals() reference the rate via decideGstTreatment() rather than the literal 0.1. Single edit point for any future AU rate change.

P3 R6. Per-invoice currency on the tender path

Add currency column to TenderContractInvoice (default 'AUD') and let the supplier pick at invoice-raise time when they're operating in another currency. Service-invoice already supports this — tender path just needs the column + the dialog field.

P4 R7. Drop redundant columns

is_cross_border and provider_gst_registered on TenderContractInvoice are derivable from gst_treatment_reason + supplier identity. Pick one source of truth (the reason string) and drop the others. Reduces "which column is canonical?" confusion.

P4 R8. Open Banking integration for confirm step

The supplier currently confirms payment receipt by eyeballing their bank statement. A future enhancement: integrate with their bank's Open Banking API (or with Stripe Treasury / Plaid) to auto-match the payment reference and pre-fill the Confirm button. Reduces dispute rate, speeds up the cycle.

9. Notes for future GST changes

If the AU government changes the GST rate

  1. After R5 is in place: edit AU_GST_RATE in invoice-template.ts, redeploy.
  2. Existing invoices keep their stored gst_amount_aud + gst_treatment_reason (snapshot at creation). New invoices use the new rate.
  3. Add an effective-date guard if needed (e.g. rate changes mid-period): check invoice created_at against rate effective dates.

If a new GST scenario emerges

For example, AU government introduces a digital-services GST treatment:

  1. Add the new scenario to computeGstTreatmentReason() — one new string and a guard.
  2. If the rate or charge decision changes, adjust decideGstTreatment().
  3. Add a unit test covering the new scenario.
  4. No schema migration needed because the treatment text is stored as a free-form string.

If a non-AU jurisdiction is added (e.g. UK VAT)

This is significantly bigger. Today gst_registered means "AU GST-registered" specifically. Multi-jurisdiction would need:

  • Per-jurisdiction registration record (e.g. UK_VAT, NZ_GST) on supplier.
  • Decision function takes jurisdiction context, not just gst_registered: boolean.
  • Rate table per jurisdiction.
  • Place-of-supply rules per jurisdiction.

For the current TalvexIT scope (AU-centric marketplace) this is over-engineering. Note for visibility only.
