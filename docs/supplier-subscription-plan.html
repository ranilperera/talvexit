1. Overview & design principles

Suppliers (individual contractors and consulting companies) sit on a four-tier ladder. Every paid tier is annual or monthly (yearly = 10× monthly), no free trial, with hard caps even on the top tier. The free tier exists so a contractor can register, finish onboarding (KYC + insurance), and publish a single listing to test the workflow — anything beyond that requires a paid plan.

Design principles

  • No trials. Trial-abuse risk on a marketplace where the supplier publishes listings is high (visibility-grab + cancel). Paid from day one.
  • Hard caps everywhere. Even Global is capped — abuse prevention > sales pitch.
  • Live counts where it matters. Concurrent-resource limits (active listings, active orders, team seats) are computed from row counts, not counters — a supplier can't bypass by waiting for a counter reset.
  • Counters only for actions. Bids placed and orders accepted are counted per period, snapshotted, then reset.
  • Anniversary-monthly reset (calendar-aware), same engine as customers.
  • Lazy rollover. No cron — the next limit check refreshes the period.

3. Quota keys (8) & how they behave

Each quota has one canonical key. The same string is referenced by the middleware (requireLimit('…')), the plan-row column, the usage panel, and any audit log entry. Two semantics buckets:

  • computed — current value is calculated from live row counts every check (no counter to reset).
  • counter — current value lives on the Subscription row and is incremented on each successful gated action; reset to 0 at every period boundary.
Quota key Plan column Semantic What "current" means
active_tasks max_active_tasks computed Count of Task rows owned by this supplier whose status is in { ACTIVE, PUBLISHED } — i.e. listings currently visible on the marketplace for browse/book.
listing_items allowed_listing_items computed Total catalogue size: every Task row owned by this supplier regardless of status (including DRAFT, PAUSED, archived). Caps the storage of catalogue items, not their visibility.
active_orders max_active_orders computed Orders this supplier is currently delivering — Order.status in { ACCEPTED, PAYMENT_HELD, IN_PROGRESS, PENDING_REVIEW, REVISION_REQUESTED }.
orders max_orders_per_month counter Subscription.current_order_count — increments when this supplier accepts an order (transitions to ACCEPTED).
active_tenders max_active_tenders computed Tenders this supplier has live submissions on — TenderBid.status = SUBMITTED against TenderRequest.status in { OPEN, REVIEWING }.
active_contracts max_active_contracts computed TenderContract.status in { PENDING, ACTIVE, IN_PROGRESS } for this supplier.
bids max_bids_per_month counter Subscription.current_bid_count — increments on every successful bid submission.
domain_categories max_domain_categories computed Length of the domains[] array on the supplier entity (ContractorProfile for solo, ConsultingCompany for org).
team_seats max_team_seats computed Active CompanyMember rows for this supplier's company (status = ACTIVE). Solo plans have 1 implicit seat (the owner).
Renamed semantics: max_active_tasks previously meant "tasks created this period (counter)." It now means "live count of active listings." Code that reads Subscription.current_task_count needs to be removed for suppliers and the column kept only for backward compatibility (or dropped in a future migration). See §14.

6. Auto-activation on registration

When a user registers as INDIVIDUAL_CONTRACTOR or ORGANIZATION_ADMIN, the registration transaction creates a Subscription row pointing at supplier-free with:

  • status = ACTIVE
  • billing_interval = MONTHLY
  • started_at = period_start = now()
  • period_end = now() + 1 calendar month (end-of-month clamped — see §7)
  • All counter columns (current_*_count) initialised to 0

This guarantees the very first call to requireLimit(...) finds an active subscription and a defined period, so the supplier can immediately attempt to publish their first listing (which the gate will allow on Free, since max_active_tasks = 1).

Where: apps/api/src/services/auth.service.ts in the registration transaction. The slug is currently hardcoded; once the supplier config moves into shared (§11), the helper AUTO_ACTIVATE_SUPPLIER_PLAN_SLUG will resolve it from config the same way AUTO_ACTIVATE_CUSTOMER_PLAN_SLUG does.

7. Period semantics & rollover

Anniversary-monthly with end-of-month clamping

A supplier who upgrades on March 15 has period boundaries on the 15th of every month thereafter. Edge cases handled by the addOneMonth() helper in SubscriptionService:

  • Jan 31 → Feb 28 (or Feb 29 in a leap year)
  • Feb 28 → Mar 28 (anniversary stabilises)
  • Mar 31 → Apr 30 → May 31 (the 31st is preserved when the target month has it)

Lazy in-service rollover

No cron. Every limit check entry point (SubscriptionService.checkLimit) calls rolloverIfDue(sub) first. If period_end <= now():

  1. Snapshot the closing period to SubscriptionUsageHistory:
    • plan_id, plan_name (denormalised so renamed plans don't lose history)
    • period_start, period_end
    • Every counter column (orders_used, bids_used, …) and its limit at the time
  2. Reset all counter columns on Subscription to 0.
  3. Compute the next anniversary boundary: period_start = period_end; period_end = period_end + 1 calendar month (clamped). Loop if multiple periods have elapsed.
  4. Write an AUDIT row of type SUBSCRIPTION_PERIOD_ROLLED.
Computed quotas need no rollover. They re-derive from row counts on every check. Only the two counter quotas (orders, bids) are snapshotted & reset.

Retention

Closed-period rows in SubscriptionUsageHistory are retained for 36 months (same as customers; USAGE_HISTORY_RETENTION_MONTHS). A daily prune job removes rows older than that. Suppliers see the current period plus the most recent 36 closed periods on their billing usage panel.

8. Where each limit is enforced

Quota Route Guard Increment
active_tasks POST /tasks requireLimit('active_tasks') — preHandler n/a (computed)
listing_items POST /tasks + PATCH /tasks/:id (when status moves out of ARCHIVED) requireLimit('listing_items') — preHandler n/a (computed)
active_orders POST /orders/:id/accept (supplier accept) and POST /orders (catalogue book) requireLimit('active_orders') — preHandler n/a (computed)
orders (per month) Same accept/book endpoints requireLimit('orders') — preHandler increment current_order_count on success
active_tenders POST /tenders/:id/bids requireLimit('active_tenders') — preHandler n/a (computed from live bid + tender state)
active_contracts POST /tender-contracts (when supplier accepts) requireLimit('active_contracts') — preHandler n/a (computed)
bids (per month) POST /tenders/:id/bids requireLimit('bids') — preHandler increment current_bid_count on success
domain_categories PATCH /contractor/profile & PATCH /companies/me requireLimit('domain_categories') — preHandler on routes that mutate domains[] n/a (computed from final array)
team_seats POST /companies/me/members/invite requireLimit('team_seats') — preHandler (already wired) n/a (computed)
The middleware is generic. makeSubscriptionGuards(svc).requireLimit(key) returns a Fastify preHandler that calls checkLimit(userId, key), returns 429 SUBSCRIPTION_LIMIT_REACHED with structured error body if the limit is hit, and increments the counter (when the key is in COUNTER_BACKED) on the way through. The trade-off: a counter increment that survives a downstream failure costs the supplier one quota slot. Acceptable for MVP — the alternative (post-success increment) requires every handler to call back into the service.

9. Trial / cancellation / downgrade

Trial

None. Every supplier plan ships trial_days = 0. Suppliers must commit to a paid tier to publish more than the free quota. Stripe checkout therefore never carries a trial_period_days for supplier plans — the customer is charged on day one.

Cancellation

Three triggers:

  1. User self-cancels via the Stripe Billing Portal → Stripe sets cancel_at_period_end = true → at current_period_end Stripe fires customer.subscription.deleted.
  2. Payment fails through the dunning grace window → Stripe terminates the sub → same webhook fires.
  3. Admin force-cancels via internal tooling → API directly cancels in Stripe → same webhook fires.

Auto-downgrade to free

On customer.subscription.deleted for a plan with plan_type matching SUPPLIER_*, the webhook handler:

  1. Looks up the supplier-free plan row.
  2. Updates the existing Subscription row (does NOT create a new one):
    • plan_id = supplier-free.id
    • status = ACTIVE
    • billing_interval = MONTHLY
    • stripe_subscription_id = null, stripe_current_period_start/end = null
    • period_start = now(), period_end = now() + 1 month
    • All current_*_count reset to 0
  3. Writes SUBSCRIPTION_AUTO_DOWNGRADED_TO_FREE audit.
Same shape as the customer auto-downgrade (customer-starter). Single code path with a per-audience plan-slug resolver — no duplication.
Behaviour change vs current: today, supplier cancellations stay CANCELLED with no auto-downgrade, which means the next gated action returns NO_ACTIVE_SUBSCRIPTION — the supplier is locked out until they resubscribe. This document changes that to mirror customer behaviour. See §14.

10. Stripe sync

The seed script's syncPlanToStripe(planId) call (already implemented for customers) is invoked for each SUPPLIER_* tier when STRIPE_SECRET_KEY is present. Per supplier plan it:

  1. Upserts a Stripe Product with id = price-id-friendly-slug.
  2. Creates two recurring Price rows: monthly (interval=month) and yearly (interval=year).
  3. For supplier-global only, also creates two USD prices (separate currency entries on the same product).
  4. Writes stripe_product_id, stripe_price_id_monthly, stripe_price_id_yearly back onto the SubscriptionPlan row.
Free tier: supplier-free never touches Stripe — no Product, no Prices, no Customer, no Subscription. The local Subscription row exists with plan_id = supplier-free.id and stripe_subscription_id = null as the marker.

11. Source of truth (proposed)

Today: supplier plan numbers live inline in apps/api/src/scripts/seed-subscription-plans.ts; customer plan numbers live in packages/shared/src/subscription-config.ts. Inconsistent.

Proposal: lift suppliers into the same shared file. The structure mirrors what we built for customers.

// packages/shared/src/subscription-config.ts

export const SUPPLIER_PLANS: Readonly<Record<string, SupplierPlanDef>> = {

14. What changes from the current implementation

Removed columns / quotas

  • max_consultant_profiles — was always returning current = 0; no enforcement; no UI; remove from schema, seed, types, and the LimitType union. Migration: drop the column.
  • max_storage_gb — same story; never enforced.
  • max_active_projects — only set on Global, no enforcement, no UI. Remove.
  • max_ai_requests_per_month on supplier plans — already not enforced for suppliers; the ai_requests column stays on the schema (customer plans use it as ai_scopes) but supplier plan rows null it out.

Renamed / repurposed quotas

  • max_active_tasks: was a monthly counter, now a live concurrent count. Implementation: drop the current_task_count read from the supplier path; replace with a prisma.task.count({ where: { user_id|company_id, status: { in: ACTIVE_STATUSES } } }) in resolveCurrentAndLimit. Counter column stays on schema for backward compat but is no longer incremented on POST /tasks.

Newly active quotas

  • allowed_listing_items: existed as a column with current = 0, now wired to prisma.task.count({ where: { user_id|company_id } }) (all statuses).
  • max_orders_per_month: existed but supplier plans had it null. Now seeded with values; gated on the supplier accept-order endpoint and the catalogue book endpoint.
  • max_active_tenders: existed but only used customer-side. Now seeded with supplier values and gated on POST /tenders/:id/bids.

Behaviour changes

  • Trial removed from every paid supplier tier (was 14 days on Solo + Company).
  • Hard caps on Global. Previously every limit was null (unlimited). Now everything is a finite number.
  • Auto-downgrade on cancel. The webhook handler no longer leaves cancelled supplier subs as CANCELLED; it auto-downgrades to supplier-free.

15. Implementation checklist

Step 1 — shared config

Create SUPPLIER_PLANS, SUPPLIER_QUOTA_KEYS, SUPPLIER_COUNTER_QUOTAS, SUPPLIER_PLAN_LIST, AUTO_ACTIVATE_SUPPLIER_PLAN_SLUG in packages/shared/src/subscription-config.ts with the numbers from §4 and the feature-flag map from §5.

Step 2 — extend LimitType

In packages/shared/src/schemas/subscription.schema.ts: add 'active_tasks' and 'listing_items' to LIMIT_TYPES. Remove 'storage' and other unused entries if they exist.

Step 3 — seed script

Refactor apps/api/src/scripts/seed-subscription-plans.ts: replace the inline supplier PLANS entries with a SUPPLIER_PLAN_SEEDS built from SUPPLIER_PLAN_LIST (mirror the existing CUSTOMER_PLAN_SEEDS mapper). Drop max_consultant_profiles / max_storage_gb / max_active_projects from the upsert payload.

Step 4 — Prisma migration

Add the migration in §14. Run prisma migrate deploy on local + every environment.

Step 5 — service layer

SubscriptionService.resolveCurrentAndLimit: add cases for active_tasks (compute via task.count with status in active set) and listing_items (task.count all statuses). Remove dead cases (storage, listing_items returning 0, consultant_profiles returning 0).

incrementUsage: drop 'tasks' from UsageType for the supplier path (kept only because customer side never used it). Confirm the only counter-backed supplier quotas left are orders and bids.

Step 6 — middleware

apps/api/src/middleware/subscription-limits.ts: update COUNTER_BACKED:

const COUNTER_BACKED = {
  // suppliers
  bids: 'bids',
  orders: 'orders',
  // customers (unchanged)
  task_bookings: 'task_bookings',
  contracts: 'contracts',
  ai_scopes: 'ai_requests',
};

Ensure active_tasks, listing_items, active_orders, active_tenders, active_contracts, domain_categories, team_seats are NOT in this map.

Step 7 — wire the new gates

Add requireLimit('active_tasks') and requireLimit('listing_items') as preHandlers on POST /tasks in task.routes.ts. Add requireLimit('active_tenders') and requireLimit('bids') on POST /tenders/:id/bids. Add requireLimit('active_orders') + requireLimit('orders') on the supplier order-accept endpoint. Add requireLimit('domain_categories') on PATCH /contractor/profile and PATCH /companies/me only when the request mutates the domains[] array.

Step 8 — auto-downgrade on cancel

apps/api/src/services/subscription-webhook.service.ts — extend the existing handleSubscriptionDeleted branch: if plan_type starts with SUPPLIER_, downgrade to supplier-free using the same shape as the customer downgrade (same audit type for parity, but log the downgrade target slug).

Step 9 — auth.service.ts auto-activation

Replace the hardcoded 'supplier-free' string with AUTO_ACTIVATE_SUPPLIER_PLAN_SLUG imported from @onys/shared.

Step 10 — usage endpoint

Extend GET /subscriptions/me/usage (currently customer-shaped) to return supplier-shaped data when the caller is on a SUPPLIER_* plan: same envelope ({ period, quotas, history }), 9 quota entries instead of 6.

Step 11 — UI: SupplierUsagePanel

Build apps/web/src/components/supplier/SupplierUsagePanel.tsx mirroring SubscriptionUsagePanel: 9 quota bars, anniversary-monthly countdown, collapsible 36-period history table. Wire it in BillingDashboard for account_type ∈ { INDIVIDUAL_CONTRACTOR, ORGANIZATION_ADMIN, COMPANY_ADMIN } when subject = 'user' or subject = 'company'.

Step 12 — UI: PlanSelector + SubscribeClient

Update the audience-aware feature builder to use the 9-quota structure for SUPPLIER cards. Same ✓/✗ rendering as customers (✗ when value = 0 with strikethrough; ✓ otherwise). Drop the diff lines (matched the customer change).

Step 13 — backfill

One-shot script: every existing Subscription on a current SUPPLIER_* plan keeps its plan but gets period_start/period_end normalised to anniversary semantics if missing, and counter columns reset only if currently above the new caps.

Step 14 — re-seed

pnpm --filter @onys/api seed:subscriptions on every environment in turn. Confirms the new prices land in Stripe (test then live).

Step 15 — manual verification

For each tier: register a fresh user → confirm auto-activation → publish listings up to the cap → confirm 429 on the cap+1 → cancel → confirm auto-downgrade to free → confirm period rollover by manipulating period_end.

16. Future change instructions

To change a supplier limit number

  1. Edit the value in packages/shared/src/subscription-config.ts.
  2. Run pnpm --filter @onys/api seed:subscriptions.
  3. No migration, no schema change. The plan-row update propagates immediately to every Subscription via the plan_id join.

To add a new quota

  1. Add the canonical key to SUPPLIER_QUOTA_KEYS and LIMIT_TYPES.
  2. Add a max_* column on SubscriptionPlan via Prisma migration.
  3. Add a case in SubscriptionService.resolveCurrentAndLimit.
  4. If counter-backed: add a counter column on Subscription, extend UsageType, add to COUNTER_BACKED, extend the rollover snapshot.
  5. Wire requireLimit('newKey') as a preHandler on the relevant route(s).
  6. Add the bar to SupplierUsagePanel and the row to PlanSelector.
  7. Re-seed.

To add a feature flag

  1. Add the boolean column to SubscriptionPlan via migration.
  2. Add to the ALL_FLAGS list in the seed script.
  3. Set the values per plan in SUPPLIER_PLANS.
  4. If the flag should be enforced: add requireFeature('flag_name') as a preHandler on the relevant route. Plan-card copy will pick up the boolean automatically.

To change pricing

  1. Edit price_aud_monthly / price_aud_yearly in shared config.
  2. Re-seed. The seed deletes old Stripe Prices (or marks them inactive) and creates new ones; stripe_price_id_* is updated on the plan row.
  3. Existing subscribers stay on their grandfathered Stripe Price. New checkouts use the new Price.

17. Open questions to confirm before implementation

  1. Is my reading of max_active_tasks vs allowed_listing_items correct? Active = currently published & visible on marketplace; listing_items = total catalogue including drafts/paused. Caps differ on Global only (25 vs 50).
  2. Domain categories cap of 28 on Global — that's exactly the size of the IT taxonomy. Is that intentional (no domain is off-limits) or should it be 29+ to leave headroom for new specialisations?
  3. Feature flags — enforce or stay decorative? Wiring requireFeature() on (e.g.) allow_video_facility would block Free + Solo from the in-call recording / KYC video features. Worth a per-flag review before wiring.
  4. USD pricing on Global only — confirmed? Or do we want USD prices on all tiers for overseas suppliers?
  5. Backfill behaviour for existing subscribers above the new caps — soft-warn for 30 days then hard-block, or hard-block immediately? Recommend the warn window since some Global subs are currently sitting on null (unlimited) and may have more than 25 active orders right now.
  6. Yearly = 10× monthly — sanity-check against business plan. Same convention as customers.
