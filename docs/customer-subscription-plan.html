2. The six quotas — what each one counts

Quota What it counts Where it's checked Type
Task bookings / month Customer clicks Book Now on a fixed-scope catalog task at /services. POST /api/v1/orders when source = 'CATALOG_BOOKING' Counter — monthly reset
Active orders Orders the customer has currently in delivery (statuses: SCOPED, ACCEPTED, IN_PROGRESS, PENDING_REVIEW, REVISION_REQUESTED). Pre-check at every action that creates a new order. Computed — live row count
Total orders / month Every order placed by the customer this period (catalog booking + AI-scope-driven + tender accepts). POST /api/v1/orders regardless of source Counter — monthly reset
AI scopes / month Each call to Generate with AI on /customer/scope. POST /api/v1/scoping/generate Counter — monthly reset
Contracts / month Tender contracts the customer signs / accepts in this period. POST /api/v1/tender-contracts Counter — monthly reset
Active tenders Tenders the customer has currently open (statuses: OPEN, INVITED, RESPONSES_RECEIVED, UNDER_REVIEW). POST /api/v1/tenders Computed — live row count

Why two task / order counters?

"Task bookings" is the catalog-booking sub-budget; "Total orders" is the aggregate. Most tiers set task_bookings ≤ total_orders so the booking cap acts as a sub-budget within the broader order cap. A customer can run out of catalog bookings but still place an AI-scope-driven order if they have headroom on Total orders.

Counter vs Computed

Counter quotas need a per-period integer column on Subscription. Increments on each successful gated action; resets to 0 at every period boundary.

Computed quotas have no counter — they're always "what's currently in flight" so we just prisma.x.count(...) at check time. Cheaper than maintaining a counter that drifts.

3. Period rollover — anniversary monthly

Spec: "the subscription quota must be reset every month from the day it started the subscription". So period boundaries follow the subscription anniversary, not the calendar month.

Rules

  • period_start is set when the subscription becomes ACTIVE (or TRIALING).
  • period_end = period_start + 1 month, calendar-aware:
    • Jan 15 10:30 → Feb 15 10:30 → Mar 15 10:30 → Apr 15 10:30 …
    • End-of-month: Jan 31 → Feb 28 (or Feb 29 in leap years) → Mar 31 → Apr 30 → May 31 … (clamps to last day of next month, then bounces back when the month allows)
  • Lazy reset. No cron job required. The rollover runs at the top of every quota check: 
    if (now >= subscription.period_end) {
  archive current period to SubscriptionUsageHistory;
  period_start = period_end;
  period_end   = period_start + 1 month;
  reset all current_* counters to 0;
}
  • If a customer doesn't transact for 3 months, their next call rolls forward 3 periods at once and archives 3 history rows in one go. Idempotent.
  • Stripe-paid plans get stripe_current_period_start / stripe_current_period_end populated by Stripe webhooks. We prefer the Stripe period when set, since billing reconciles to Stripe's reality. For free / trial plans, our locally-computed period is authoritative.

New columns on Subscription

// Quotas 2 (active orders) and 6 (active tenders) are computed live — no counter
Existing counter cleanup — the schema currently has current_task_count, current_project_count, and current_bid_count, all rooted in supplier-side semantics. We rename current_task_count → current_task_publish_count (supplier-side) and add a separate current_task_booking_count for the customer side, so the two flows don't share a counter.

New table: SubscriptionUsageHistory

Archives every closed period so the customer can see what they used and when, even after plan upgrades.

model SubscriptionUsageHistory {
  id                       String       @id @default(cuid())
  subscription             Subscription @relation(fields: [subscription_id], references: [id])
  subscription_id          String
  plan_id                  String       // snapshotted — plan can change between periods
  plan_name                String       // snapshotted human label
  period_start             DateTime
  period_end               DateTime
  task_bookings_used       Int          @default(0)
  task_bookings_limit      Int?         // null = unlimited at the time of the period
  orders_used              Int          @default(0)
  orders_limit             Int?
  ai_scopes_used           Int          @default(0)
  ai_scopes_limit          Int?
  contracts_used           Int          @default(0)
  contracts_limit          Int?
  archived_at              DateTime     @default(now())

  @@index([subscription_id, period_end])
  @@map("subscription_usage_history")
}

Why snapshot the limits? A customer who upgrades mid-period needs accurate historical numbers later. If we only stored "used", a future audit wouldn't know whether 8 orders against a 10-cap was within plan or not at the time.

7. Pre-block UX — warn before the wall

Every quota-checking endpoint returns the live numbers in its success response, so the frontend can show a warning before the next click:

// POST /api/v1/orders — successful response after consuming a quota slot
{
  "success": true,
  "data": { "id": "ord_...", ... },
  "subscription_usage": {
    "task_bookings": { "used": 4, "limit": 5,  "remaining": 1 },
    "orders":        { "used": 4, "limit": 5,  "remaining": 1 },
    "active_orders": { "used": 2, "limit": 2,  "remaining": 0 }
  }
}
  • The frontend caches this in its react-query store and shows a banner when remaining ≤ 1 — "Heads up — 1 task booking left this period".
  • At click-time on Book Now / Generate with AI / Create Tender / Create Contract, the button label shifts to a "warning" variant when at the cap, and the click handler shows a confirm dialog before posting:
    "Booking this task will use your last task booking for this period. Your quota resets on Feb 15 (12 days). Upgrade to keep going."
  • When the cap is already at zero, the button is disabled with a tooltip: "Task bookings limit reached — upgrade or wait until your period resets on Feb 15."

8. Subscription renewal flow

TriggerWhat happens
Free plan, period boundary reached during a quota check Lazy rollover: archive period → reset counters → bump period_start / period_end. No external action; status stays ACTIVE.
Paid plan, Stripe invoice.payment_succeeded Existing handler in subscription-webhook.service.ts updates stripe_current_period_*. We extend it to also archive the closed period to SubscriptionUsageHistory and reset counters.
Paid plan, Stripe invoice.payment_failed Status → PAST_DUE. Quota checks fail with a different error code (SUBSCRIPTION_PAST_DUE) so the UI can show a "fix billing" CTA rather than an "upgrade" CTA. Period rollover paused until payment succeeds.
Paid plan, Stripe customer.subscription.deleted Status → CANCELLED, but we leave the customer's existing access alive until the end of the current paid period (read cancel_at_period_end on Stripe). At period end, downgrade them to customer-starter automatically and let the lazy-rollover handle the next period.
Trial expiry (Stripe customer.subscription.trial_will_end 3 days before) Email reminder; no platform action.
Trial converts to active Status TRIALING → ACTIVE. No quota reset — the trial period continues seamlessly into the first paid period.
User upgrades plan in /customer/billing Stripe upgrade → webhook → update plan_id on Subscription. Counters carry over (don't reset; the customer paid for the period continuation). New limits apply to the same period.
User downgrades plan Same flow, but a downgrade can put the customer over-cap on some quotas. We don't claw back; only block new actions until usage falls below the new cap.

9. Auto-activation on customer signup

Already partly in place. In apps/api/src/services/auth.service.ts at registration:

  1. Customer registers via POST /api/v1/auth/register with account_type = 'CUSTOMER'.
  2. Inside the registration transaction, the service looks up the plan with slug customer-starter.
  3. If found, creates a Subscription row: plan_id, user_id, status = ACTIVE, started_at = now(), period_start = now(), period_end = now() + 1 month (anniversary).
  4. If not found (seed not run), logs a warning and continues — registration doesn't fail. Ops fixes the seed.
  5. The customer's first quota-gated action sees status = ACTIVE and a fresh period; rollover logic handles every subsequent reset.

Q4 detail — soft-cap warning thresholds

Different per plan because a 20%-of-25 warning (5 remaining) makes sense for Enterprise but doesn't for Free Starter (a 20%-of-2 warning would be before the user does anything).

function shouldWarn(planSlug: string, remaining: number, limit: number | null): boolean {
  if (limit === null) return false; // unlimited — never warn
  // Higher tiers: warn at 20% headroom
  if (planSlug === 'customer-professional' || planSlug === 'customer-enterprise') {
    return remaining <= Math.ceil(limit * 0.2);
  }
  // Free + Business: warn at the last slot
  return remaining <= 1;
}

14. Instructions to change limits in the future

Customer-side limits, prices, trial days, plan names, and warning thresholds all live in one TypeScript file. No schema migration is needed for any of these changes — the columns already exist on SubscriptionPlan; you're just changing the values stored on the rows. Re-running the seed upserts by slug, so existing subscription rows automatically pick up the new limits next time they hit a quota check.

14.1 To change a quota limit

Example: bump Business task bookings from 5 to 8.

  1. Open packages/shared/src/subscription-config.ts.
  2. Find 'customer-business' and edit limits.task_bookings: 
    'customer-business': {
  // ...
  limits: {
    task_bookings: 8,   // was 5
    active_orders: 2,
    orders:        5,
    ai_scopes:     1,
    contracts:     1,
    active_tenders: 1,
  },
},
  3. Rebuild the shared package and re-run the seed: 
    pnpm --filter shared build
pnpm --filter @onys/api seed:subscriptions
  4. That's it. Customers already on Business pick up the new limit on their next quota check (no re-subscribe / re-deploy required for them).

14.2 To change a price

Example: change Professional from $99/mo → $79/mo.

  1. In subscription-config.ts, edit price_aud_monthly: 
    'customer-professional': {
  // ...
  price_aud_monthly: 79,
  price_aud_yearly: 790,   // remember the 10× monthly convention
  // ...
}
  2. Re-run the seed. If STRIPE_SECRET_KEY is set, the seeder syncs the new Stripe Prices automatically (creating new Price records on Stripe — Stripe Prices are immutable so the seeder creates a new one and writes the new ID back to the plan row). The old Price ID stays on Stripe for any in-flight subscriptions; new subscribers get the new price.
  3. Existing subscribers stay on the old price until they upgrade / change interval, which matches Stripe's standard subscription billing convention.

14.3 To add a new quota

This does require a schema migration. Steps:

  1. Add the column to SubscriptionPlan (e.g. max_priority_support_minutes) and a counter to Subscription (e.g. current_priority_support_minutes) in apps/api/prisma/schema.prisma.
  2. Generate a migration: cd apps/api && pnpm prisma migrate dev --name add_priority_support_minutes.
  3. Add the limit type key to LIMIT_TYPES in packages/shared/src/schemas/subscription.schema.ts.
  4. Add the field to CustomerPlanLimits in subscription-config.ts and populate it on every plan.
  5. Add a case to resolveCurrentAndLimit in SubscriptionService (counter-backed reads from sub.current_*; computed reads via this.prisma.x.count()).
  6. If counter-backed: add to COUNTER_BACKED in middleware/subscription-limits.ts and to fieldMap + UsageType in SubscriptionService.incrementUsage + rolloverIfDue.
  7. Wire requireLimit('priority_support_minutes') into the relevant route's preHandler.
  8. Add to the customerQuotas array in SubscriptionService.getCustomerUsage so the billing page picks it up.
  9. Add a label to QUOTA_LABELS + QUOTA_ORDER in SubscriptionUsagePanel.tsx for the UI display.

14.4 To change the warning thresholds

Edit the warningThresholdFor() helper in subscription-config.ts. Currently:

export function warningThresholdFor(planSlug: string, limit: number | null): number {
  if (limit === null || limit === 0) return 0;
  const isHigherTier =
    planSlug === 'customer-professional' || planSlug === 'customer-enterprise';
  if (isHigherTier) {
    return Math.max(1, Math.ceil(limit * 0.2));
  }
  return 1;
}

The function returns the threshold count (an integer) — the UI compares remaining ≤ threshold and flips into the warn state. No DB change; the next page load picks it up.

14.5 To change the period length (e.g. quarterly instead of monthly)

Edit the addOneMonth() helper in apps/api/src/services/subscription.service.ts — replace the setUTCMonth(+1) with the desired interval. The migration SQL in 20260506120000_customer_subscription_rebuild uses Postgres INTERVAL '1 month' for backfill; that one is one-shot and won't matter going forward. Stripe-driven billing for paid plans is already controlled by Stripe's price interval, so the local helper only matters for free / cancelled plans.

14.6 To change retention on usage history

Edit USAGE_HISTORY_RETENTION_MONTHS in subscription-config.ts. There's no prune job written yet (history just accumulates) — when one is added, it should read this constant.

14.7 To change the auto-activated free plan

Set auto_activate_on_signup: true on a different plan in subscription-config.ts and ensure exactly one plan has it. The AUTO_ACTIVATE_CUSTOMER_PLAN_SLUG export reads this and the registration flow in auth.service.ts uses the slug. The hook there is currently hard-coded to customer-starter; if you change which slug is auto-activated, also update auth.service.ts:106-114.

TL;DR for limit changes: Edit one TypeScript file → run two pnpm commands → done. No schema migration, no front-end deploy, no Stripe portal click-through (the seed handles Stripe sync if its key is set). Existing subscribers pick up the new limits on their next quota check.
