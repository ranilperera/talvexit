TalvexIT — Email configuration & customisation

v1.0 · 2026-05-15 · How the platform sends every system email — transport, queue, templates, links
TL;DR. Every system email is sent through Microsoft Graph (no SMTP, no SendGrid). The API enqueues a typed job onto a BullMQ queue named email; the workers process consumes it, picks the matching build* template function in apps/workers/src/jobs/email.worker.ts, and posts to Graph. To change copy: edit the build* function. To add a new email: add a variant to EmailJobPayload + a builder + a case in the worker switch. Sender identity (EMAIL_FROM, Azure tenant + client + secret) lives in apps/workers/.env.

§1 The data flow

Top to bottom, where email-related code lives and who reads it:

1

Trigger — service-layer code in apps/api/src/services/*

Any business event that should notify a user calls emailQueue.add(type, payload). Examples: auth.service.ts registers a user and enqueues verify-email; order.service.ts queues order-accepted on acceptance.

2

Queue producer — BullMQ 'email' Redis queue

The API creates new Queue('email', { connection: redisConnection() }) in apps/api/src/app.ts (line 121) and injects it into every service that needs to send mail. Worker-side schedulers (insurance expiry, order SLA, etc.) create their own producers pointing at the same queue name.

3

Worker consumer — apps/workers/src/jobs/email.worker.ts

One new Worker<EmailJobPayload>('email', …, { concurrency: 5 }) consumes the queue. A big switch (data.type) routes each payload to the matching build* function, which returns { to, subject, html }.

4

Transport — graphEmailService.sendEmail(...)

Lives in apps/workers/src/services/graph-email.service.ts. Acquires an OAuth2 client-credentials token (cached, refreshed 60s before expiry), then POSTs to https://graph.microsoft.com/v1.0/users/{EMAIL_FROM}/sendMail. Returns on HTTP 202.

5

Recipient inbox

Delivered from EMAIL_FROM (default alerts@onsys.com.au) with display name EMAIL_FROM_NAME. Any link inside the body is an absolute URL built from FRONTEND_URL via apps/api/src/utils/urls.ts.

§2 Transport — Microsoft Graph API

File: apps/workers/src/services/graph-email.service.ts  single sender no SMTP

The platform does not use nodemailer, SendGrid, Mailgun, SES, or SMTP. All email goes through the Microsoft Graph API using OAuth2 client-credentials. The class is small (~165 lines):

  • getAccessToken() — POSTs to https://login.microsoftonline.com/{tenant}/oauth2/v2.0/token with scope https://graph.microsoft.com/.default, caches the token, refreshes 60s before expiry.
  • sendEmail({ to, subject, html, replyTo?, attachments? }) — builds a Graph message body, POSTs to /v1.0/users/{EMAIL_FROM}/sendMail with saveToSentItems: true. Returns silently on 200/202, throws on anything else.
  • Exports a singleton graphEmailService + a convenience sendEmail(opts) function the worker calls directly.
Implications of this choice.
  • The sending mailbox must be a real Azure AD user (or shared mailbox) the app registration has been granted Mail.Send permission for.
  • There is no per-tenant rate limiter in the code — Graph itself enforces ~10,000 messages / 24h per mailbox. Bulk sends will need throttling if we ever approach that.
  • Attachments use base64 (contentB64) embedded in the JSON body — Graph's hard limit is 4 MB per message including all attachments. Anything bigger must use the upload-session pattern (not implemented).
  • HTML only. The text field on SendEmailOptions is unused downstream — we don't currently send multipart/alternative.

§4 Queue layer — BullMQ email queue

Queue name is the literal string 'email' everywhere. There is one logical queue with five concurrent consumers (workers, default concurrency).

Producers (services that enqueue jobs)

FileWhat it enqueues
apps/api/src/app.ts:121The main API producer — injected into every service constructor that needs to send mail
apps/workers/src/jobs/credential-purge.job.ts:24Worker-side scheduled purge notifications
apps/workers/src/jobs/insurance-expiry.scheduler.ts:15"Your insurance expires in N days" warnings
apps/workers/src/jobs/order-sla.scheduler.ts:15Order SLA breach reminders
apps/workers/src/jobs/service-invoice-overdue.scheduler.ts:10Service-invoice overdue reminders

Service-layer call pattern (from AuthService.register()):

await this.emailQueue.add('verify-email', {
  type:       'verify-email',
  to:         user.email,
  verify_url: emailUrls.verifyEmail(verificationToken),
  userId:     user.id,
});

The first argument is the BullMQ job name (used for metrics & logs). The type field on the payload is what the worker switches on — they're conventionally the same string but only type is load-bearing.

Consumer

Single Worker in apps/workers/src/jobs/email.worker.ts (~1,470 lines). Concurrency 5. On failed it logs the error; BullMQ then retries with the queue's default backoff. There is no manual removeOnComplete/removeOnFail config — failed jobs persist for inspection via BullMQ Board.

§5 Templates — the single source file

File: apps/workers/src/jobs/email.worker.ts  ~1,470 lines  44 templates

Every template lives in this one file. There is no templating engine (Handlebars / MJML / EJS) and no external HTML files. Each template is an inline tagged template literal returning { to, subject, html }.

Anatomy of the file

  1. Lines 7–217 — EmailJobPayload union. The discriminated union typing every job payload. Each variant has a literal type string + the fields that template needs. Adding a new template starts here.
  2. Lines 219–1306 — build* functions. One per template. Returns { to: { email }, subject: string, html: string }. All use teal #0d9488 branding and the brand name talvexIT inline.
  3. Line 1307 — escape(). Minimal HTML escaping. Use it on any user-supplied string that goes into the template body to prevent injection.
  4. Lines 1315–1462 — the switch (data.type). Routes each payload to the right builder. The default branch logs an unknown-type warning and silently drops the job — do not rely on this for validation.

What a builder looks like

function buildVerifyEmail(to: string, verify_url: string) {
  return {
    to:      { email: to },
    subject: 'Verify your talvexIT account',
    html: `
      <div style="font-family:sans-serif;max-width:480px;margin:auto">
        <h2 style="color:#0d9488">Welcome to talvexIT!</h2>
        <p>Click the button below to verify your email address.</p>
        <a href="${verify_url}"
           style="display:inline-block;padding:12px 24px;background:#0d9488;
                  color:#fff;border-radius:6px;text-decoration:none;font-weight:bold">
          Verify Email
        </a>
        <p style="color:#666;font-size:13px;margin-top:24px">
          This link expires in 24 hours. If you didn't create an account, ignore this email.
        </p>
      </div>
    `,
  };
}
Inline CSS — on purpose. Email clients (Gmail, Outlook) strip <style> blocks and external stylesheets unpredictably. Every visual attribute lives inline. Don't try to factor styles into a shared CSS file — it will silently break in Outlook.

§7 Special case — login OTP

Login OTP is the one email type with state outside the queue payload. The 6-digit code and its expiry / attempt counter live in the EmailOtpChallenge table, managed by apps/api/src/services/email-otp.service.ts:

  • createOtpChallenge(prisma, userId, meta) generates a random 6-digit code, hashes it, persists the challenge, and returns the plaintext for one-time delivery.
  • verifyOtpChallenge(prisma, challengeToken, code) validates with constant-time comparison, increments attempt counter, marks the challenge VERIFIED on success.
  • Service-layer rate-limit: 3 sends per 15 min per user. Route-layer anti-spray-rate-limit: 5 sends per 15 min per IP on /auth/resend-otp.
  • TEST_BYPASS_OTP=true env var skips the entire OTP step — never set this in production. Used by automated E2E tests.

§8 Common customisation recipes

Edit the copy of an existing email

  1. Open apps/workers/src/jobs/email.worker.ts.
  2. Find the matching build* function (e.g. buildVerifyEmail). Use Ctrl+F on the subject line to jump to it.
  3. Edit the subject and/or the inline-HTML html template literal. Keep CSS inline.
  4. Restart the workers process. The API does not need a restart — it only enqueues; the worker holds the template.

Change the sender name or address

  1. Pick a new mailbox in Azure AD (must be a real user the Graph app has Mail.Send for).
  2. Update EMAIL_FROM and EMAIL_FROM_NAME in apps/workers/.env on the VM.
  3. Restart the workers process.
  4. Send a test (see §9).

Rotate the Azure secret

  1. In the Azure portal, App Registrations → the email app → Certificates & secrets → New client secret.
  2. Copy the new value before navigating away — it's only shown once.
  3. Update AZURE_EMAIL_CLIENT_SECRET in apps/workers/.env.
  4. Restart the workers process. The next getAccessToken() call will pick up the new secret.
  5. After verifying mail is flowing, delete the old secret in the portal.

Add a new email template

  1. Define the payload variant. In email.worker.ts, add a new branch to the EmailJobPayload union with a unique type string and the fields you'll interpolate: 
    | { type: 'task-flagged-for-review'; to: string; task_title: string; reviewer_url: string }
  2. Write the builder. Add a buildTaskFlaggedForReview(...) function that returns { to, subject, html }. Match the visual style of neighbours (teal heading, button, expiry note). Use escape(...) on any user-supplied string.
  3. Wire it into the switch. Add a case 'task-flagged-for-review': branch that calls buildTaskFlaggedForReview(data) and breaks.
  4. Enqueue from a service. In the API service that triggers it: 
    await this.emailQueue.add('task-flagged-for-review', {
  type:         'task-flagged-for-review',
  to:           reviewer.email,
  task_title:   task.title,
  reviewer_url: emailUrls.companyMembers(),
});
  5. Type-check both packages: pnpm --filter @onys/api typecheck + pnpm --filter @onys/workers typecheck.
  6. Restart the workers process and test.
The EmailJobPayload union is duplicated in the API service code where each service declares its Queue<EmailJobPayload> typing — the worker has its own copy. Keep them in sync; a typo in the API-side type means the worker switch won't know about your new type and will silently drop the job.

Stop a specific notification temporarily

Two options, in order of preference:

  1. Comment out the emailQueue.add(...) call in the service that triggers it. The cleanest cutoff — no half-rendered jobs end up in Redis.
  2. Make the worker case a no-op. In the switch: 
    case 'order-cancelled':
  // TEMPORARILY DISABLED — see ticket #XYZ
  break;
    Useful when you can't deploy the API but can deploy workers, or when the source is external (e.g. a Stripe webhook).

Add a brand-wide change (logo, footer, palette)

There is no central layout helper. Every template embeds its own HTML wrapper. A brand-wide change today means a find-and-replace across all 45 build* functions. If we do a serious rebrand we should first extract a wrap(headerHtml, bodyHtml, footerHtml) helper and refactor each builder to call it — see §11.

§9 Testing emails locally

Send a real email to your own inbox

  1. Make sure apps/workers/.env has the Azure creds + EMAIL_FROM set.
  2. Make sure Redis is running (docker-compose up redis).
  3. Start the workers: pnpm --filter @onys/workers dev
  4. Trigger the flow from the UI (e.g. register a new account to test verify-email).
  5. Watch the worker console — you'll see [email] Processing job type="verify-email" to="…" then [graph-email] Sent to … on success.

Inspect queue state

BullMQ jobs persist in Redis. Use redis-cli or BullMQ Board to inspect:

# list queued
redis-cli --raw KEYS 'bull:email:*' | head

# completed / failed counts
redis-cli LLEN 'bull:email:completed'
redis-cli LLEN 'bull:email:failed'

# requeue a failed job (BullMQ Board UI — preferred)

Preview a template without sending

There is no built-in preview server. Quick approaches:

  • Copy the html string into an HTML file and open it in a browser.
  • Use a separate inbox-test service (Litmus, Email on Acid) for Outlook / Gmail / Apple Mail preview matrices.
  • Send to a Mailtrap / Mailpit sandbox inbox in dev by pointing EMAIL_FROM at a tester address you control. (The Graph transport itself can't be redirected to a local SMTP catcher.)

§10 Operations — monitoring, retries, failures

ConcernHow it's handled
Retries on transient failures BullMQ default retry policy (exponential backoff). No per-job retry config in the worker constructor — we rely on the queue defaults.
Unknown job type Switch falls through to default, logs [email] Unknown job type: X, returns. The job is marked completed (not retried). Fix: add the missing case.
Graph 4xx (e.g. invalid recipient) sendEmail throws → the BullMQ job moves to failed. Will retry; permanent 4xx will eventually land in the failed list for manual inspection.
Graph 429 (rate-limited) Currently bubbles up as a thrown error — retries follow. If we hit sustained rate limits we'll need an explicit backoff aware of the Retry-After header.
Bad sender mailbox Graph returns 403 / 404 — the job lands in failed. Check EMAIL_FROM and the app registration's Mail.Send grant.
Token expiry Cache auto-refreshes 60 s before expiry. No action needed.
Audit trail The OAuth event and every send are logged to the workers stdout. The corresponding business event (e.g. USER_REGISTERED) is written to AuditLog by the API before the email is enqueued.

§11 Rebrand & follow-ups

Known follow-ups after the 2026-05 rebrand.
  • EMAIL_FROM=alerts@onsys.com.au still uses the legacy onsys.com.au domain. After the Onsys → Waveful operator rename, this should move to a Waveful or TalvexIT-branded mailbox (e.g. alerts@talvexit.com) once the new mailbox is provisioned in Azure AD and the app registration is granted Mail.Send on it.
  • Templates hard-code the brand name "talvexIT" and the teal #0d9488 across all 45 builders. The default fallback in graph-email.service.ts for EMAIL_FROM_NAME is still onys.online.
  • There is no shared layout helper. Extract a wrap() helper before the next visual refresh.

§12 Anti-patterns — don't do these

  • Don't call sendEmail(...) directly from API code — always go through the queue. The API process has no Graph credentials and shouldn't block on outbound HTTP.
  • Don't embed user-supplied strings into html without passing through escape(). A name containing <script> becomes a tracking pixel in Gmail.
  • Don't hard-code http://localhost:3000 in a template — always use emailUrls.* so the link works in every environment.
  • Don't add an <style> block or <link> stylesheet — Gmail / Outlook strip them. Inline CSS only.
  • Don't introduce a new transport library (nodemailer, SendGrid SDK, AWS SES SDK) without a deliberate decision. The single-transport choice is what makes auditing & rate-limiting tractable.
  • Don't commit AZURE_EMAIL_CLIENT_SECRET to git. It's a send-as credential for the company mailbox.
  • Don't set TEST_BYPASS_OTP=true in production. It disables the OTP step entirely.
  • Don't rely on the default branch in the worker switch as validation — it silently drops jobs. Add an explicit case for every type you enqueue.
