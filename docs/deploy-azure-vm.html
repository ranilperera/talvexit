Deploy to Azure VM

Date:
2026-05-06
Codebase:
talvex-v1 @ main
Target:
API VM (vm-aueast-onsys-web-04) at 10.1.3.28 behind HAProxy at 10.1.3.100
Audience:
Whoever runs the deploy. Assumes shell access via SSH and read-write on /opt/talvex/app.

Two phases: first-time setup (only run once per VM) and rolling deploy (every release). Skip Phase A if the VM is already provisioned and running.

Phase A — First-time setup

A1. Generate the secrets

On any machine with openssl:

echo "JWT_ACCESS_SECRET=$(openssl rand -base64 48)"
echo "JWT_MFA_SECRET=$(openssl rand -base64 48)"
echo "MFA_ENCRYPTION_KEY=$(openssl rand -base64 32)"

Save these in your password manager / Key Vault. Each must be ≥ 32 chars; the API refuses to boot in production with a missing or weak secret.

Lose MFA_ENCRYPTION_KEY and every MFA-enabled user has to re-enrol. The key cannot be recovered or guessed. Back it up to a separate Key Vault before you deploy.

A2. Get an ABR GUID

Register at abr.business.gov.au/Tools/WebServices. Free; takes a few minutes. The GUID lets the API call ABN lookups against the official Australian Business Register. Without it, every customer hitting the "Verify ABN" button gets a 500 with code ABR_NOT_CONFIGURED.

A3. SSH to the API VM and lay out the directory

ssh onsyswebuser@vm-aueast-onsys-web-04
sudo mkdir -p /opt/talvex/app
sudo chown $(whoami):$(whoami) /opt/talvex/app
cd /opt/talvex/app
git clone https://github.com/ranilperera/talvexit-v1.git .

A4. Create .env.prod

nano /opt/talvex/app/.env.prod
chmod 600 /opt/talvex/app/.env.prod

Minimum required contents:

# ── Core ─────────────────────────────────────────────────────────────────
NODE_ENV=production
DATABASE_URL=postgresql://<user>:<pass>@<db-host>:5432/onys_prod
REDIS_URL=redis://onys_redis:6379

# ── Public site URL (canonical host, no trailing slash) ─────────────────
FRONTEND_URL=https://talvexit.com
NEXT_PUBLIC_SITE_URL=https://talvexit.com
NEXT_PUBLIC_API_URL=https://talvexit.com
API_URL=http://onys_api:3001

# ── Auth secrets (from step A1 — 32+ chars, never the dev defaults) ─────
JWT_ACCESS_SECRET=<paste from A1>
JWT_MFA_SECRET=<paste from A1>
MFA_ENCRYPTION_KEY=<paste from A1>

# ── HAProxy front (proxy IP for rate-limit + IP-aware features) ─────────
TRUSTED_PROXY=10.1.3.100

# ── CORS — only the exact prod origin, no trailing slash ────────────────
CORS_ORIGIN=https://talvexit.com

# ── ABR (from step A2) ──────────────────────────────────────────────────
ABR_GUID=<paste your ABR GUID>

# ── Azure Blob (existing) ───────────────────────────────────────────────
AZURE_STORAGE_CONNECTION_STRING=<your-connection-string>
AZURE_STORAGE_CONTAINER=onys-files

# ── Stripe (paid subscriptions) ─────────────────────────────────────────
STRIPE_SECRET_KEY=sk_live_...
STRIPE_WEBHOOK_SECRET=whsec_...

# ── Email (Azure Communication Services or your provider) ──────────────
AZURE_EMAIL_CONNECTION_STRING=<your ACS connection string>
EMAIL_FROM=no-reply@talvexit.com

# ── Search-console verification (optional but recommended) ─────────────
GOOGLE_SITE_VERIFICATION=
BING_SITE_VERIFICATION=

# ── Hard-fail guard: must NOT be 'true' in prod ─────────────────────────
# TEST_BYPASS_OTP=  ← leave unset
The API container's validateEnv refuses to boot if any of DATABASE_URL, JWT_ACCESS_SECRET, JWT_MFA_SECRET, MFA_ENCRYPTION_KEY, FRONTEND_URL, CORS_ORIGIN, ABR_GUID is missing. By design — silent fallback to dev defaults is worse than a clear startup crash.

A5. Initial database setup

cd /opt/talvex/app

# Build the API image so we can run prisma against it
docker compose -f docker-compose.prod.yml build api

# Apply every migration in apps/api/prisma/migrations/
docker compose -f docker-compose.prod.yml run --rm api \
  pnpm --filter api prisma migrate deploy

# Seed the 8 subscription plans (4 customer + 4 supplier).
# With STRIPE_SECRET_KEY set, this also creates Stripe Products + Prices
# and writes the IDs back to the plan rows.
docker compose -f docker-compose.prod.yml run --rm api \
  pnpm --filter api seed:subscriptions

A6. First start

docker compose -f docker-compose.prod.yml up -d
docker compose -f docker-compose.prod.yml ps

You should see onys_api, onys_web, onys_redis all Up and healthy.

A7. HAProxy is already configured

Per the existing /etc/haproxy/haproxy.cfg, the backend portal1_api_backend already points at 10.1.3.28:3001 with option forwardfor. No HAProxy changes needed for new deploys — the Docker network is the only thing the API container is exposed on.

A8. Sanity check

# From the VM:
curl -s http://localhost:3001/api/v1/health
# Expected: {"status":"ok","timestamp":"..."}

# From outside:
curl -s https://talvexit.com/api/v1/health

Phase B — Rolling deploy (every release)

The four-line ritual every time main advances:

ssh onsyswebuser@vm-aueast-onsys-web-04
cd /opt/talvex/app

# 1. Pull
git pull origin main

# 2. Migrations (idempotent — does nothing if no new ones)
docker compose -f docker-compose.prod.yml run --rm api \
  pnpm --filter api prisma migrate deploy

# 3. Plan seed (idempotent — upsert by slug; safe to run every time)
docker compose -f docker-compose.prod.yml run --rm api \
  pnpm --filter api seed:subscriptions

# 4. Build + restart
docker compose -f docker-compose.prod.yml up -d --build api web
Order matters. Migrations and seed must run before the new API container starts, or you'll hit "column does not exist" on first request. The --rm spawn pattern uses the new image but exits before the long-lived container starts.

If you want it scripted, the existing deploy.sh already does most of this — add the seed line before the up step.

What gets exercised on boot

LayerWhat happens
apps/api/src/app.ts validateEnv Hard-fails if JWT_ACCESS_SECRET, JWT_MFA_SECRET, MFA_ENCRYPTION_KEY, ABR_GUID, CORS_ORIGIN, FRONTEND_URL, or DATABASE_URL is missing in prod.
Trust proxy TRUSTED_PROXY env makes Fastify read req.ip from X-Forwarded-For (HAProxy at 10.1.3.100). Without it, all rate-limit buckets share one IP and lock the whole site after 20 auth requests.
prisma migrate deploy Applies any outstanding migrations — append-only AuditLog trigger, customer subscription rebuild (period columns + counters + history), schema cleanup migrations.
seed:subscriptions Upserts the 8 plans from packages/shared/src/subscription-config.ts. With Stripe key set, syncs Products / Prices.
Web container env NEXT_PUBLIC_SITE_URL, JWT_ACCESS_SECRET (passed to web so middleware can verify JWT in edge runtime via jose), Stripe publishable key.

Rollback

cd /opt/talvex/app
git log --oneline -10        # find the last good commit
git checkout <good-sha>
docker compose -f docker-compose.prod.yml up -d --build api web
Migrations don't auto-rollback. Prisma migrations are forward-only. If a migration is the problem, you need to write a corrective migration. Don't run prisma migrate resolve --rolled-back unless you've manually undone the schema change with SQL first.

Things to verify post-deploy

CheckHow
API booted docker compose ... logs api | tail -50 — should see [startup] FRONTEND_URL: https://talvexit.com and no env-var errors.
Migrations applied docker compose ... run --rm api pnpm --filter api prisma migrate status — every migration "Applied".
Plans seeded Hit https://talvexit.com/api/v1/subscriptions/plans — should return 8 plans.
ABN verify works Login as a test user, go to /customer/profile, enter a real AU ABN → it should hit ABR and populate the entity name.
Customer registration Register a new customer → check the DB: 
SELECT id, plan_id, period_start, period_end
FROM subscriptions WHERE user_id = '<new user id>';
Should return one row with the Free Starter plan and a period 1 month out.
Web JWT verify Open /customer/dashboard in a private window — middleware should verify the JWT (no spoofable cookie). Manually setting onys_account_type=PLATFORM_ADMIN in DevTools should NOT bypass the role gate.

Common failures & fixes

SymptomLikely causeFix
[startup] Missing required env vars: ... in API logs, container exits .env.prod missing or one of the required keys is unset Edit /opt/talvex/app/.env.prod, add the missing key, docker compose ... up -d
Every API call returns 429 "Rate limit exceeded" TRUSTED_PROXY not set — Fastify keys all rate-limit buckets to HAProxy's IP Add TRUSTED_PROXY=10.1.3.100 to .env.prod and restart the API container
CORS errors in browser console after deploy CORS_ORIGIN has a trailing slash (e.g. https://talvexit.com/) Browsers send Origin without trailing slash. Strip it. The CORS middleware also strips it defensively but env hygiene is cleanest.
Customer can't register — 500 error The column current_task_booking_count does not exist Migration not applied before API container started Run prisma migrate deploy from step B2, then restart the API container
ABR_NOT_CONFIGURED on every ABN verify attempt ABR_GUID missing from .env.prod Get a GUID from Step A2, paste, restart
Stripe webhook events all return 401 INVALID_SIGNATURE STRIPE_WEBHOOK_SECRET in .env.prod doesn't match the secret on Stripe's webhook endpoint config Stripe Dashboard → Developers → Webhooks → click your endpoint → "Reveal" the signing secret → paste into .env.prod
Web middleware passes everyone, even spoofed account types JWT_ACCESS_SECRET not set in the web container's env Confirm docker-compose.prod.yml threads JWT_ACCESS_SECRET into the web service environment block. Restart.
Once you've done one full first-time setup and one rolling deploy, the day-to-day shape is just: SSH → git pull → migrate deploy → seed → build & up. ~3 minutes for a typical release.
